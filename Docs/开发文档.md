## 1. 项目概述

UniSky 是一个基于 Unity 的模拟飞行项目。核心目标是把 X-Plane 11 的 Boeing 737-800 机模文件和北京大兴国际机场（ZBAD）地景转换为 Unity 可编辑资源，在此基础上构建完整的飞行模拟系统，最终完成一次完整的五边飞行（起飞、爬升、巡航、进近、着陆）。

飞机各组件的运动逻辑与真实飞机及 X-Plane 11 中的呈现一致——能动的组件必须能动，运动逻辑一致，有动画的也要有动画。飞行数据来自 JSBSim 物理引擎，Unity 负责渲染和玩家操作界面，JSBSim 后台负责飞行物理计算。

### 分步路线图

| 阶段 | 内容 | 状态 |
|------|------|------|
| 第一步 | 机模与地景导入、部件运动动画、涂装切换（不含仪表） | **当前阶段** |
| 第二步 | JSBSim 完整接入、玩家操作、五边飞行 | 待开始 |
| 后续 | 仪表系统、音效系统 | 待规划 |

## 2. 整体架构

### 2.1 子系统划分

Unity 主场景包含五个子系统：

| 子系统 | 第一步 | 说明 |
|------|------|------|
| 飞机系统 | 实现 | 模型、部件驱动、动画、涂装 |
| 地景系统 | 实现 | 机场 OBJ 资源 + Cesium 兜底 |

| 音效系统 | 暂跳过 | 后续规划 |

**Floating Origin Manager** 是全局组件，所有子系统的 Transform 都通过它偏移，保证大坐标下的浮点精度。

```

│       │                                               │
│       ▼                                               │
│  ┌─────────────┐                                      │
│  │  输入管理器  │  ←── 第一步：键盘测试数据模拟输入    │
│  └─────────────┘                                      │
│       │  UDP 控制指令（第二步接入）                   │
│       │  第一步直接喂测试值                           │

│  └─────────────┘                                      │
│       │                                               │
│       ▼                                               │
│  ┌──────────────────┐                                 │
│  │    部件驱动层     │  驱动 FBX 动画曲线 / Transform  │
│  └──────────────────┘                                 │
│                                                       │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────┐   │
│  │  地景系统 │  │  相机系统 │  │ Floating Origin  │   │
│  └──────────┘  └──────────┘  └──────────────────┘   │
└──────────────────────────────────────────────────────┘
         ▲  控制指令（UDP）        结果数据（UDP）  ▲
         └──────────────────────────────────────────┘
                   JSBSim.exe（后台独立进程）
                   （第二步接入，第一步暂不连接）
```

### 2.3 数据流说明

- 玩家在 Unity 里操作键盘/鼠标
- Unity 将操作转换为控制指令（如方向舵偏转值），通过 **UDP** 发送给后台运行的 JSBSim
- JSBSim 计算物理结果（如方向舵实际偏转角、飞机姿态、速度等），通过 **UDP** 返回给 Unity
- Unity 根据返回数据驱动飞机各部件的动画和位移
- **JSBSim 是唯一数据源**，玩家输入只是发给 JSBSim 的控制指令，不直接控制部件

## 3. 技术选型与理由

### Unity 2022.3.62f3c1 LTS + URP

使用长期支持版本保证稳定性。URP（Universal Render Pipeline）已在项目初始化时配置好。

### XPlane2Blender + Blender（机模导入）

X-Plane OBJ 文件内含 `ANIM_rotate` / `ANIM_trans` 指令，直接记录每个部件的旋转轴、平移范围和对应 dataref。这些 dataref 就是运动逻辑的语义来源——例如：

```
ANIM_rotate  0 1 0   0.00 30.00   0 1.0   sim/flightmodel2/wing/aileron1_deg[4]
```

表示副翼绕 Y 轴，随 `aileron1_deg[4]` 从 0° 旋转到 30°。

用 XPlane2Blender 插件导入后，这些 ANIM 数据会变成 Blender 的 Action 关键帧，导出 FBX 时动画曲线一并带走，进 Unity 后直接有 AnimationClip 可用，无需手工还原运动逻辑。

其他方案（手动逐步导入、只导静态 OBJ）都会丢失这个映射关系，工作量是此方案的 3 倍以上。

### JSBSim（飞行物理引擎）

JSBSim 是开源飞行动力学模型（FDM），提供精确的飞行物理计算。采用 **UDP Socket** 接入方式，JSBSim 作为独立进程后台运行：

- 业界主流做法，参考微软 AirSim 架构
- JSBSim 官方直接支持网络输出，配置简单
- 两端解耦，可独立调试
- 本机 UDP 通信延迟在 1ms 以内，感觉不到

不选择编译为 DLL Native Plugin 的原因：在 Windows 上把 JSBSim C++ 源码编译为 DLL 配置复杂，跨平台和 C++/C# 互操作踩坑时间可达 1-2 天，收益（零延迟）在模拟飞行场景下完全感觉不到。

### Cesium for Unity（地形兜底）

机场资源里没有覆盖到的区域（外围地形、城市建筑、远景）由 Cesium for Unity 提供。Cesium 提供全球地形高程和卫星底图，作为背景填充层。参考：https://cesium.net.cn/learn/unity/unity-quickstart/

### Floating Origin

解决 Unity 单精度浮点数在大坐标下精度不足（约 7 位有效数字，飞行 5km 后坐标超过 5000，精度已不足 1mm）导致的飞机组件抖动问题。超过 1000m 触发整体平移，把场景原点重置到飞机附近。参考：https://manuel-rauber.com/2022/04/06/floating-origin-in-unity/

## 4. 数据映射层（关键设计）

### 4.1 两套数据体系的区别

X-Plane OBJ 的 ANIM 动画指令由 `sim/flightmodel2/*` 系列 dataref 驱动，这是 **X-Plane 的部件显示数据**（表示部件当前应显示的状态，单位通常是度或 0~1 比例）。

JSBSim 输出的字段是 `fcs/*`、`propulsion/*`、`gear/*`，这是 **JSBSim 的飞行力学数据**（单位和命名体系与 X-Plane 不同）。

**两套体系不能直接对接**，中间必须有一个 DataMapper 做换算和映射。

### 4.2 DataMapper 架构

```
JSBSim 输出（fcs/elevator-pos-deg = 15.0）
      │
      ▼
DataMapper.cs
  - 根据字段名取值
  - 做单位换算（如 n1% → 弧度/秒）
  - 输出 flightmodel2 语义的数值
      │
      ▼
部件驱动层（elevator1_deg = 15.0 → 驱动升降舵 Transform）
```

### 4.3 第一步的简化方案

第一步不接 JSBSim，DataMapper 的输入端接键盘测试数据：

```csharp
// 第一步：测试模式，键盘直接设置映射值
DataMapper.Set("elevator1_deg", Input.GetAxis("Vertical") * 25f);

// 第二步：接入 JSBSim 后，DataMapper 从 UDP 数据包读取
DataMapper.Set("fcs/elevator-pos-deg", jsbSimData["fcs/elevator-pos-deg"]);
```

部件驱动层的代码第一步和第二步完全一样，只需在第二步补齐 DataMapper 的 JSBSim 输入端。

## 7. Floating Origin

### 7.1 原理

Unity 使用单精度浮点数（float）表示世界坐标，约 7 位有效数字。当飞机飞离原点 5000m 后，坐标值为 5000.xxx，精度约为 0.001m（1mm）。随着距离增大，精度快速下降，最终导致飞机各组件出现抖动。

Floating Origin 的解决方案：当飞机距世界原点超过阈值（1000m）时，把场景里所有根节点的位置整体平移，使飞机重新回到原点附近，浮点精度始终保持在最高水平。从飞机视角看，世界没有变化。

### 7.2 实现要点

- `FloatingOriginManager.cs` 挂在场景根节点，每帧检查飞机与原点的距离
- 超过 1000m 时，对所有注册的根节点（地景、Cesium 地形、环境对象等）统一执行偏移
- 飞机本身不偏移（或重置为原点），相对位置不变
- Cesium for Unity 需要特殊处理：通过 `CesiumGeoreference` 的 API 同步偏移
- 参考实现：https://manuel-rauber.com/2022/04/06/floating-origin-in-unity/

### 7.3 注册机制

```csharp
// 地景、Cesium、粒子系统等根节点都需注册
FloatingOriginManager.Instance.Register(airportRoot);
FloatingOriginManager.Instance.Register(cesiumGeoreference);
```

## 9. 涂装系统

### 9.1 原理

B737-800 机身使用一张主贴图（`738fuselage.dds`）控制外观。涂装切换就是在运行时把 Livery 节点上的材质贴图替换为对应涂装目录下的贴图，不需要重新加载模型。

### 9.2 涂装目录

```
Assets/Aircrafts/B737-800/Model/Liveries/
├── ChinaSouthern/   ← 默认涂装（中国南方航空）
├── AirChina/        （中国国际航空）
├── Alaska/
├── Delta/
├── KLM/
├── Qantas/
└── Southwest/
```

原始贴图来源：`XPlaneAssets/Aircraft/Boeing B737-800/liveries/` 各子目录下的 PNG/DDS 文件。

### 9.3 LiveryManager.cs

```csharp
// 切换涂装示例
LiveryManager.Instance.SwitchLivery("ChinaSouthern");
```

运行时替换 `Livery` 节点 MeshRenderer 上的 `_BaseMap` 贴图属性即可。

## 10. 输入系统

### 10.1 键位映射（草案，边做边完善）

| 按键/操作 | 含义 | 发送给 JSBSim 的指令 |
|------|------|------|

| 鼠标 Y 轴（激活后） | 俯仰输入 | `fcs/elevator-cmd-norm` |
| A / D | 偏航（方向舵） | `fcs/rudder-cmd-norm` |
| 上 / 下方向键 | 油门增减 | `fcs/throttle-cmd-norm` |
| F | 襟翼收一档 | `fcs/flap-cmd-norm`（减小） |
| G（长按）| 襟翼放一档 | `fcs/flap-cmd-norm`（增大） |
| G（单击）| 起落架收放 | `gear/gear-cmd-norm` |
| 右键单击 | 激活/取消鼠标控制操纵杆 | —— |
| 数字键 1/2/3 | 切换相机视角 | —— |

> 以上为初始草案，实际开发中会根据操作体验边做边调整。

### 10.2 InputManager.cs 职责

第一步：读取键盘/鼠标输入，直接把测试值送给 DataMapper（跳过 JSBSim），验证部件动作。

第二步：读取键盘/鼠标输入，通过 UDP 发送给 JSBSim，接收 JSBSim 返回数据，送给 DataMapper。

两步之间 InputManager 内部逻辑只需切换数据来源，DataMapper 和部件驱动层不变。

## 13. 坐标系与单位注意事项

### X-Plane OBJ 坐标系

- 格式标识：文件头部 `A 800 OBJ`（OBJ8 格式）
- 单位：米
- 坐标系：**右手系**，Z 轴朝后（X 右、Y 上、Z 朝后）
- 注意：与 Unity 的左手系（Z 朝前）不同，导入时需做坐标系转换

### Unity 坐标系

- 坐标系：**左手系**，Y 轴朝上，Z 轴朝前
- 坐标系转换通常在 Blender 导出 FBX 时处理（Forward=-Z、Up=Y）
- 导入后检查飞机朝向是否为 Z 轴正方向（朝前）

### 贴图格式

- X-Plane 原始贴图为 DDS 格式（部分为 PNG）
- Unity 对 DDS 支持有限，建议批量转为 PNG 后导入
- `_NML` 后缀为法线贴图，`_LIT` 后缀为夜间发光贴图，导入时注意在 Inspector 中设置正确的贴图类型

### Git LFS

项目已配置 Git LFS，FBX、OBJ、DDS、PNG 等大文件通过 LFS 管理，克隆时需确认 `git lfs install` 已执行，并运行 `git lfs pull` 拉取大文件。
